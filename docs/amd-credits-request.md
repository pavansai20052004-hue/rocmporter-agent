# AMD Developer Cloud credits — outreach kit

Goal: GPU access so ROCmPorter can run **execution** validation (build + test
migrated code on real AMD hardware) and attach a hardware receipt to every
migration PR. Today we can only prove *compile*.

## Email — ready to send

**Subject:** `ROCmPorter (AMD hackathon) — requesting Developer Cloud credits for ROCm CI validation`

```
Hi <name>,

I built ROCmPorter during the AMD hackathon and have kept working on it since.
It scans any GitHub repository for CUDA lock-in, scores its ROCm readiness with
line-level evidence, and opens a pull request that migrates the code to HIP.

It's live and open source (MIT):
  App:  https://rocmporter-agent.vercel.app
  Repo: https://github.com/pavansai20052004-hue/rocmporter-agent

The part I think is relevant to AMD: the migration engine is deterministic
first. A curated CUDA→HIP mapping pass converts the mechanical majority with no
model involved at all — files it fully converts never touch an LLM — and only
the semantic remainder goes to a model, grounded in curated ROCm documentation.
Every migrated file reports its provenance, so a reviewer can see exactly what
was mechanical and what was AI-assisted.

Output is already compile-verified: CI runs hipcc inside AMD's official
rocm/dev-ubuntu-22.04 container on every push, and the example in the repo
(examples/vector_add) is machine-generated by the engine and compile-checked
continuously. That badge on the README is a real build.

What I can't do yet is prove the migrated code *runs*. Compile-verified is not
execution-verified, and I'd rather say that plainly than imply otherwise.

That's my ask: would it be possible to get AMD Developer Cloud credits (or GPU
access in any form) for this project? With a GPU runner I would:

  - register it as a self-hosted GitHub runner and run the migrated test suites
    on real AMD hardware
  - attach a "builds and passes tests on ROCm" receipt to every migration PR
  - publish the results openly, including anything that fails

The reason I think this is worth AMD's while: every repository this tool
migrates is a repository that can then run on AMD instead of only NVIDIA. The
bottleneck for most teams isn't willingness, it's the porting effort. That's
exactly what this removes — and hardware-verified receipts are what would make
teams actually trust it.

Happy to share a walkthrough, the migration engine internals, or a case study
on whichever repository would be most useful to you.

Thanks for the hackathon — it's what got this built.

Best,
GJV Pavansai
<your email> · <your LinkedIn or GitHub>
```
